# Insurellm RAG Assistant

A fully local, Retrieval-Augmented Generation (RAG) system that answers natural-language questions about a fictional insurance company ("Insurellm") using its internal knowledge base. Built around Ollama for local inference/embeddings and ChromaDB for vector storage, with an LLM-driven ingestion pipeline and a two-pronged evaluation harness (retrieval metrics + LLM-as-judge).

## Why this exists

Most toy RAG demos split documents with a fixed-size text splitter and call it done. This project pushes a bit further on both ends of the pipeline:

- **Ingestion** uses an LLM to *semantically* chunk each document (headline + summary + original text) rather than blindly cutting on character count.
- **Retrieval** rewrites the user's question, retrieves with both the original and rewritten query, merges the candidates, and asks an LLM to rerank them before truncating to the final top-k.
- **Evaluation** doesn't just eyeball answers — it computes retrieval-quality metrics (MRR, nDCG, keyword coverage) against a 150-question labeled test set, *and* runs an LLM judge to score generated answers on accuracy, completeness, and relevance.

## Architecture

```
Ingestion pipeline (ingest.py)
  knowledge-base/*.md
        │
        ▼
  LLM semantic chunking (headline + summary + original_text)
        │
        ▼
  HuggingFace embeddings (all-MiniLM-L6-v2)
        │
        ▼
  ChromaDB (preprocessed_db/)

Query pipeline (answer.py)
  user question
        │
        ▼
  LLM query rewrite ──────┐
        │                 │
        ▼                 ▼
  retrieve (k=20)    retrieve (k=20)   [original query]  [rewritten query]
        │                 │
        └──────┬──────────┘
               ▼
             merge 
               ▼
          LLM rerank
               ▼
          top-k (k=10)
               ▼
          LLM answer generation
```

## Features

- **Local-first**: defaults to `ollama/llama3.2` for generation/reranking/judging and `all-MiniLM-L6-v2` for embeddings — no API keys required out of the box.
- **Swappable models**: `litellm` is used as the completion layer, so swapping in a hosted model (OpenAI, Groq, etc.) is a one-line change.
- **Structured outputs everywhere**: chunking, reranking, and the LLM judge all use Pydantic-validated structured responses instead of fragile string parsing.
- **Two Gradio apps**:
  - `app.py` — chat UI with a side-by-side panel showing the retrieved context for each answer.
  - `evaluator.py` — a dashboard to run retrieval and answer evaluations across the full test set, with color-coded metrics and per-category bar charts.
- **Quantitative + qualitative evaluation**: MRR / nDCG / keyword coverage for retrieval, and LLM-judge scoring (accuracy, completeness, relevance) for generated answers, both broken down by question category (`direct_fact`, `temporal`, `spanning`, `comparative`, `numerical`, `relationship`, `holistic`).

## Tech stack

| Layer | Tool |
|---|---|

## Prerequisites

- Python 3.12+ (the ingestion code uses nested-quote f-strings that require 3.12)
- [Ollama](https://ollama.com/download) installed and running locally
- The `llama3.2` model pulled:
  ```bash
  ollama pull llama3.2
  ```
- A `knowledge-base/` directory of markdown files, organized into subfolders by document type (each subfolder name becomes the document's `type` metadata)

## Usage

**1. Ingest the knowledge base** (run once, or whenever source docs change):
```bash
python pro_implementation/ingest.py
```
This chunks every `.md` file under `knowledge-base/` using an LLM, embeds the chunks, and writes them into `preprocessed_db/`.

**2. Chat with the assistant:**
```bash
python app.py
```
Opens a Gradio chat window with retrieved context shown alongside each answer.

**3. Run a single evaluation from the CLI:**
```bash
python evaluation/eval.py <test_row_number>
```
Prints retrieval metrics and an LLM-judged answer score for one test case.

**4. Run the full evaluation dashboard:**
```bash
python evaluator.py
```
Runs all 150 retrieval and answer evaluations, with progress bars, color-coded summary metrics, and per-category bar charts.

## Evaluation methodology

**Retrieval** (per question, averaged across its keywords):
- **MRR** — reciprocal rank of the first chunk containing each keyword
- **nDCG@10** — discounted gain for keyword presence across the top 10 retrieved chunks
- **Keyword coverage** — % of expected keywords found anywhere in the retrieved set

**Answer quality** (LLM-as-judge, 1–5 scale, scored against a human-written reference answer):
- **Accuracy** — factual correctness (any wrong fact caps the score at 1)
- **Completeness** — whether all information from the reference answer is present
- **Relevance** — whether the answer stays on-topic without padding

Both are broken down by question category so you can see, for example, whether the system struggles more with `comparative` or `holistic` questions than `direct_fact` ones.

## Configuration

Key tunables, all near the top of `pro_implementation/answer.py`:

| Variable | Default | Purpose |
|---|---|---|
| `MODEL` | `ollama/llama3.2` | Model used for rewrite/rerank/answer generation |
| `RETRIEVAL_K` | 20 | Chunks pulled per query (original + rewritten, each) |
| `FINAL_K` | 10 | Chunks kept after LLM reranking |
